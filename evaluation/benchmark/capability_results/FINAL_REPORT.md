# Deep Research Agent — 三架构能力上限横向对比报告

> 3 个问题 × 3 种架构 × 3 轮重复 = 27 次完整运行
> 总耗时 150 分钟 · 总 Token ~10M · 总成本 ~$6

## ⚠️ Limitations

1. **事实正确性** — 本评估测量报告的方法论质量（结构完整性、来源支撑度、不确定性诚实度），**不直接测量事实正确性**。未做 NLI 或人工事实核实。
2. **Judge 模型** — Coverage 和 Honesty 评分使用 DeepSeek v4-pro（与生成报告相同的模型），存在自评偏差风险。三种架构在这两个维度的得分（0.89-0.99, 4.3-4.8）大部分处于自评噪声范围内，**不应作为区分性论据**。
3. **样本量** — 3 题 × 3 轮 = 每架构 9 个数据点。不足以进行统计显著性检验。结论为方向性指示。
4. **React Q2 引用解析限制** — React Agent 在 Q2 的 3 轮中产出的 `## 来源` 使用了 `[N] Author, "Title", Year. URL` 格式，当前引用解析器尚未兼容此格式。Q2 React 的 corroboration 指标和 `unique_domains_cited` 因此系统性偏低，**实际值应更高**。
5. **N/A 字段** — `cross_perspective_pct` 为 Multi-Agent 特有架构能力，Pipeline/ReAct 标记 N/A。`fulltext_ratio` 对 Multi-Agent 和 ReAct 不适用（搜索发生在 Agent 内部，不经过 Tavily extract 路径），标记 N/A。

---

## 核心结论：无银弹

**三种架构在整体复合分上无实质差距（pipeline 0.740*, multi-agent 0.642, react 0.633）。真正有意义的信息不在排名里，在差异化特征里。**\n\n> \*Pipeline 均值基于 8/9 有效轮次（Q1 R3 因 TavilyKeyPool bug 崩溃被排除）。

```
            Pipeline              Multi-Agent           React
            ────────              ───────────           ─────
最适合      确定型任务            多视角任务             探索型任务
核心优势    强印证率最高(8.7%)    声明数最高(38.6)       诚实度最高(4.8)
            结构最完整            覆盖度最高(0.99)       最快(101s)
                                                       最便宜($0.02)
                                                       探索广度最高(Q3: 12.3域名)

核心代价    有稳定性bug(1/9轮crash) 成本中($0.25)       强印证率最低(2.7%)
            成本高($0.38)         某些题目方差大         引用格式不稳定
```

**选择取决于任务特征，而非某个架构"更好"。**

## 架构差异化特征（有数据支撑）

### Pipeline
- **最适合**：问题结构清晰、搜索路径可预测的任务
- **强项**：交叉验证最严格（强印证 8.7%）、结构最完整（多节报告格式）
- **弱项**：成本随报告长度线性增长（Q3 报告长 → judge token 1.5M → $0.90）
- **稳定性**：方差最小（σ pipeline=0.03-0.05，Q1 除外）

### Multi-Agent
- **最适合**：需要多个独立视角、子问题间无强依赖的任务
- **强项**：声明数最高（38.6，多 Agent 独立提取）、覆盖度最高（0.99）、强印证率 3.9%
- **弱项**：对比型（0.549）和开放型（0.625）上碎片化损害连贯性
- **注意**：`cross_perspective_pct` 是 Multi-Agent 独有架构能力，但实测值很低（0.000-0.026）——跨 Agent 印证检测词的词法方法在当前数据上几乎未触发

### React
- **最适合**：研究路径不清晰的开放探索型任务
- **强项**：速度最快（101s）、成本最低（$0.025）、诚实度最高（4.8）、Q3 探索广度最高（12.3 域名）
- **弱项**：引用格式不稳定（Q1/Q3 ✅，Q2 ❌）；过早停止搜索（27k tokens/run，仅为 Pipeline 的 5%）
- **关键特征**：token 用量是 Pipeline 的 1/20——不是因为快，而是因为搜得少。这是质量-效率的明确 tradeoff，不是纯粹的"优势"

## 已知残余问题

| # | 问题 | 影响 | 优先级 |
|---|------|------|--------|
| 1 | **TavilyKeyPool `record_error()` 参数 bug** — 调用时使用了不存在的 `is_quota` 关键字参数。Pipeline Q1 R3 因此崩溃 | 1/27 轮数据丢失，Pooled 客户端无法正确处理 quota 错误 | 🔴 下次修 |
| 2 | React `## 来源` 格式不稳定——Q1/Q3 能解析，Q2 的 `[N] Author, Title, Year. URL` 格式无法解析 | Q2 React 的 corroboration/domains 指标偏低 ~0.10-0.15 composite | 🟡 下次修 |
| 3 | LLM-as-Judge 自评天花板——coverage 0.89-0.99, honesty 4.3-4.8，几乎无区分度 | 35% 权重（coverage+honesty）不可靠 | 🟡 引入独立 judge |
| 4 | `fulltext_ratio` 对所有架构为 0 或 N/A——Tavily extract 路径的 content_type 字段未正确设置 | process metrics 表缺失此数据 | 🟢 低 |
| 5 | Cross-perspective corroboration 实际触发率接近 0（词法方法太保守） | Multi-Agent 独有指标未能体现 | 🟢 低 |
| 6 | 样本量 9 点/架构——不足以做统计检验 | 结论为方向性指示 | 🟢 低 |
