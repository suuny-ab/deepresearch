# Deep Research Agent v0.1.1 Acceptance Report

Date: 2026-06-11

## 1. Scope

This report records acceptance results for the Deep Research Agent v0.1.1 improvement release.

v0.1.1 was designed to improve feedback, observability, and failure clarity after the v0.1.0 MVP exposed product-quality issues during real usage.

Relevant documents:

- v0.1.0 design spec: `docs/superpowers/specs/2026-06-10-deep-research-agent-design.md`
- v0.1.0 implementation plan: `docs/superpowers/plans/2026-06-10-deep-research-agent.md`
- v0.1.1 improvement spec: `docs/superpowers/specs/2026-06-11-deep-research-agent-v0.1.1-improvements.md`

## 2. Summary Verdict

| Area | Result |
|---|---|
| v0.1.1 improvement acceptance | Passed |
| Offline automated tests | Passed |
| CLI missing-key error path | Passed |
| CLI invalid argument probe | Passed |
| Online `--verbose` workflow observability | Passed |
| Online successful research report generation | Failed |

Overall conclusion:

```text
v0.1.1 passes its intended engineering/improvement acceptance criteria.
It does not yet pass product-level acceptance for reliably generating a successful online research report.
```

The main remaining product issue is report generation success rate. In online runs, the tool can complete the workflow and save an artifact, but the generated report may fail validation because the writer output does not satisfy the validator's citation contract.

## 3. v0.1.1 Acceptance Criteria

The v0.1.1 spec defines acceptance as follows:

1. All existing tests still pass.
2. New tests for failure reports, failed filenames, CLI failure messaging, and verbose summaries pass.
3. A validation-failure report is clearly labeled as failed.
4. A validation-failure filename ends in `-failed.md`.
5. CLI output distinguishes success from validation failure.
6. CLI progress output is no longer misleading.
7. `--verbose` provides enough intermediate detail for a user to confirm that planning, searching, synthesis, writing, and review occurred.
8. No external API calls are made in default tests.

## 10. Known Issue: Successful Online Report Generation Rate

The key unresolved issue is not workflow execution or observability. The unresolved issue is successful report generation rate.

Observed behavior:

- The workflow reaches planning, search, synthesis, writing, review, and saving.
- Tavily returns search results.
- The writer generates a report-like output.
- The validator rejects the report because body claims do not include acceptable citations before the `Sources` section.

Root product issue:

```text
The writer output format and validator citation contract are not aligned.
```

Likely causes:

1. The model may use numeric references such as `[1]` while URLs only appear in `## Sources`.
2. The validator currently requires allowed URLs to appear in the body before `## Sources`.
3. The writing prompt may not be strict enough about inline Markdown links or direct body URLs.
4. There is no automatic rewrite attempt after validation failure.

This should be addressed in a separate design discussion for v0.1.2 or v0.2.

## 11. Recommended Next Design Topic

Recommended topic:

```text
Improve online successful report generation rate by aligning the citation contract between writer prompt and validator.
```

Possible directions:

1. Accept numeric citations like `[1]` when they map to allowed URLs in `## Sources`.
2. Strengthen writer prompt to require inline Markdown links in every key claim.
3. Add one automatic rewrite attempt after validation failure.
4. Add debug/trace artifact for the rejected raw report to diagnose writer behavior safely.

## 12. Final Acceptance Statement

```text
v0.1.1 improvement acceptance: Passed.
Online successful research report acceptance: Failed.
```

Expanded statement:

```text
v0.1.1 satisfies its intended engineering and observability goals:
- tests pass,
- validation failures are clearly labeled,
- failed reports use -failed.md,
- CLI distinguishes success and failure,
- progress output is node-aware,
- --verbose exposes workflow internals,
- default tests do not call external APIs.

However, the product is not yet reliable at producing successful online research reports. The next iteration should focus on citation contract alignment and retry strategy.
```
