# Deep Research Agent v0.1.1 验收报告

日期：2026-06-11

## 1. 范围

本报告记录了 Deep Research Agent v0.1.1 改进版本的验收结果。

v0.1.1 旨在改进反馈、可观测性和失败清晰度，因为 v0.1.0 MVP 在实际使用过程中暴露出产品质量问题。

## 2. 总结裁决

| 领域 | 结果 |
|---|---|
| v0.1.1 改进验收 | 通过 |
| 离线自动化测试 | 通过 |
| CLI 缺失密钥错误路径 | 通过 |
| CLI 无效参数检测 | 通过 |
| 在线 `--verbose` 工作流可观测性 | 通过 |
| 在线成功研究报告生成 | 失败 |

总体结论：

```text
v0.1.1 通过了预期的工程/改进验收标准。
但尚未通过产品级验收，无法可靠生成成功的在线研究报告。
```

主要剩余产品问题是报告生成成功率。在在线运行中，工具可以完成工作流并保存产物，但生成的报告可能无法通过验证，因为写入器输出不满足验证器的引用契约。

## 3. v0.1.1 验收标准

v0.1.1 规格定义了以下验收标准：

1. 所有现有测试仍然通过。
2. 关于失败报告、失败文件名、CLI 失败消息和详细摘要的新测试通过。
3. 验证失败报告清楚标记为失败。
4. 验证失败文件名以 `-failed.md` 结尾。
5. CLI 输出区分成功和验证失败。
6. CLI 进度输出不再具有误导性。
7. `--verbose` 提供足够的中间细节，让用户确认规划、搜索、综合、写作和审查已发生。
8. 默认测试中不进行外部 API 调用。

## 10. 已知问题：在线报告生成成功率

关键未解决问题不是工作流执行或可观测性，而是成功报告生成率。

观察到的行为：

- 工作流到达规划、搜索、综合、写作、审查和保存阶段。
- Tavily 返回搜索结果。
- 写入器生成类似报告的输出。
- 验证器拒绝报告，因为正文声明在 `Sources` 部分之前不包含可接受的引用。

根本产品问题：

```text
写入器输出格式与验证器引用契约不一致。
```

可能原因：

1. 模型可能使用数字引用（如 `[1]`），而 URL 仅出现在 `## Sources` 中。
2. 验证器当前要求允许的 URL 出现在 `## Sources` 之前的正文中。
3. 写作提示可能对嵌入式 Markdown 链接或直接正文 URL 要求不够严格。
4. 验证失败后没有自动重写尝试。

这应该在 v0.1.2 或 v0.2 的单独设计讨论中解决。

## 11. 推荐的下一个设计主题

推荐主题：

```text
通过对齐写入器提示和验证器之间的引用契约，提高在线成功报告生成率。
```

可能方向：

1. 接受数字引用（如 `[1]`），当它们映射到 `## Sources` 中的允许 URL 时。
2. 加强写入器提示，要求每个关键声明都有嵌入式 Markdown 链接。
3. 在验证失败后添加一次自动重写尝试。
4. 为被拒绝的原始报告添加调试/跟踪产物，以安全诊断写入器行为。
