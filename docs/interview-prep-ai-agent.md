## 三、项目深挖：Deep Research Agent

这是你整场面试的核心话题。以下按简历上的每个技术点，都准备了 **"面试官会怎么问"** + **"你怎么回答"**。

### 3.1 项目总览

> **一句话定位**：一个自动分解研究问题、检索网络信息、交叉验证证据并生成引用报告的多模式 AI Agent 系统。

**面试官可能问：** "介绍一下你这个项目整体是做什么的？"

**回答要点：**
- 输入：用户提一个研究问题（比如"AI 搜索引擎的发展趋势"）
- 系统做的事：分解子问题 → 搜索（Tavily）→ 提取声明 → 交叉验证 → 写报告
- 输出：带 `[N]` 编号引用的结构化学术风格报告
- 三种执行模式可切换：Pipeline（快速） / Multi-Agent（全面） / ReAct（灵活）

### 3.2 三种 Agent 架构设计

> **简历原文**：实现了三种差异化的执行模式——Pipeline（确定性 5 节点工作流）、Multi-Agent（子问题级并行 Agent + Coordinator 合并与跨 Agent 印证）、以及 ReAct V2（带反思循环和结构化记忆的自主 Agent）。

**面试官可能问：** "为什么做三种架构？不嫌多吗？"

**你的回答：**
- 动机：不同的使用场景需要不同的 trade-off。Pipeline 最快（确定性强、延迟低），Multi-Agent 最全面（每个子问题有独立 Agent，不会互相影响），ReAct 最灵活（可以自主调整研究路径）。
- 技术实现：三种模式共享同一套 `ResearchState` 状态模型和提示词体系。
- 关键数据：FACT 验证中 Multi-Agent 准确率 92.3%，远高于 Pipeline 的 22.6%——因为 Coordinator 阶段的跨 Agent 印证能显著提升引用准确性。

**追问 1：** "Multi-Agent 是怎么并行跑的？"
- Python `ThreadPoolExecutor` 实现子问题级并发，每个 Agent 独立执行 search → extract → validate。`Coordinator` 汇总所有 Agent 输出，检测跨 Agent 的矛盾点和印证点。

**追问 2：** "ReAct V2 的'反思增强'具体怎么做的？"
- 在标准 ReAct 循环（Think → Act → Observe）基础上，每 N 轮插入一个 Reflection 步，LLM 评估当前进展、识别信息缺口、动态调整研究计划。
- 关键设计：三阶段架构——Phase 0 生成研究计划，Phase 1 反思循环，Phase 2 将笔记转化为证据卡片和报告。
- 这解决了"盲目搜索直到饱和"的问题，变成"有策略地填补知识缺口"。

**追问 3：** "三种模式怎么切换的？"
- CLI 参数 `--architecture pipeline|multi-agent|react`。在 `runner.py` 中根据参数选择不同的 graph builder 函数构建对应的 StateGraph。

### 3.3 两阶段证据流水线（核心亮点）

> **简历原文**：将事实提取与交叉验证拆分为两个独立阶段，解决单次 LLM 调用同时承担"发散提取"和"收敛验证"导致的保守输出问题。

**面试官可能问：** "两阶段证据流水线的 insight 是什么？"

**你的回答：**
- 核心洞察：**提取和验证是冲突的认知任务。** 提取需要发散（尽量抓取潜在相关的信息），验证需要收敛（严格判断是否被独立来源印证）。让 LLM 一次完成这两个任务，倾向于输出保守的结果——要么只提取明显相关的信息，要么过早判断无印证。
- 做法：Phase 1 只做提取（1 次 LLM 调用，不加验证指令），Phase 2 按子问题并行做交叉验证（N 次 LLM 调用）。
- 后验证阶段：用代码层面的域名多样性检查和 URL 有效性检查，覆盖 LLM 输出的不可靠性问题。

**追问 1：** "交叉验证的级别怎么定义的？"
- `single_source`：只有一个来源
- `weakly_corroborated`：多个来源提及相同信息，但域名关联度高
- `strongly_corroborated`：3 个以上独立域名的来源印证同一事实

**追问 2：** "如果所有来源都是同一域名，怎么处理？"
- 筛选阶段有域名多样性约束：每个子问题每个域名最多一条，防止单一域名的信息被过度权重。

### 3.4 引用合规系统

> **简历原文**：设计 7 维度 [N] 引用校验器（缺失来源章节 / 裸 URL / 重复编号 / 未定义引用 / 非法 URL / 未使用来源 / 重复来源），失败时自动构造诊断信息并触发一次重写。

**面试官可能问：** "引用校验器为什么需要 7 种失败模式？"

**你的回答：**
- LLM 输出的引用格式可能出错的方式太多了——不按顺序引用、引用不存在的编号、在正文里直接贴 URL、来源部分定义重复编号——每种都需要不同的处理方式。
- 采用**优先级依次检查**的策略：先检查最严重的错误（缺失 Sources 部分 > 裸 URL > 重复编号...），一旦发现就停止并返回具体的诊断信息。
- 自动重写机制：校验失败时，把失败原因和允许的 URL 列表注入到重写 prompt 中，让 LLM 在下一轮生成合规的报告。
- 结果：3/3 测试问题通过，`100%` 引用合规率。

**追问 1：** "为什么不在 prompt 里一次性要求好？"
- 做过实验。LLM（即使是 DeepSeek）在长上下文中稳定输出引用格式是一个开放问题。与其赌 prompt 能一次搞定，不如设计一个验证-重试的闭环——这是更可靠的工程解法。

**追问 2：** "FACT 引用真实性验证是什么？"
- FACT 验证不是检查格式，而是检查**内容真实性**：用 Jina AI 爬取被引用 URL 的网页内容，然后用 LLM 逐条核验报告的声明是否被网页内容支持。
- 这个发现了架构间的显著差异：Multi-Agent 92.3% 准确率，Pipeline 只有 22.6%。原因是 Pipeline 的搜索范围更窄，容易引用不直接支持声明的页面。

### 3.5 评估体系

> **简历原文**：15 个零 LLM 确定性评估指标（来源利用率、印证率、引用密度等），支持架构间 ablation 对比和 LangSmith 全链路可观测性。

**面试官可能问：** "为什么评估体系很重要？"

**你的回答：**
- LLM 应用的评估是公认的难题。我的思路是：**用确定性代码替代 LLM-as-judge**，避免"用一个 LLM 评估另一个 LLM"带来的偏差和成本。
- 15 个指标覆盖：引用合规率、来源利用率（用了多少搜索结果）、印证分布（single / weak / strong 的比例）、引用密度（每段引用数）等。
- 这些指标集成了 LangSmith，每次运行自动记录，支持架构间对比（Ablation Study）。

**追问 1：** "为什么不用 LLM 做评估？"
- LLM-as-judge 有两个问题：1) 偏好问题——LLM 倾向于给更长的回答更高分；2) 成本问题——每次评估都要多一次 LLM 调用。15 个确定性指标加起来不到 1 毫秒运行时间，零成本。

### 3.6 工程化实践

**面试官可能问：** "项目工程化方面有什么亮点？"

- **218 个离线单元测试**：完全离线，使用 FakeLLMClient（返回预配置响应）和 FakeSearchClient（返回桩数据），保证 CI 环境下不依赖外部 API。
- **Protocol 级依赖注入**：`LLMClient` 和 `SearchClient` 都是 Python Protocol，测试时直接注入 fake 实现，Prod 时注入 DeepSeek/Tavily 真实客户端。
- **按阶段追踪 Token 成本**：`TokenUsage` 模型记录每个节点的 token 消耗，可以精确知道哪个阶段最贵。
- **TTL 持久化缓存**：搜索层集成 TTL 缓存，减少重复 API 调用，节省成本。
- **指数退避重试**：LLM 调用层用 tenacity 实现指数退避重试，应对 API 抖动。

**追问 1：** "哪些是你自建的，哪些用了 LangSmith？"
- LangSmith 做了链路追踪和评估集成。最开始我自建了 30% 的平台基础设施（日志、追踪），后来发现这些不是核心竞争力，删掉了——集中精力做 Agent 质量本身。这是一个重要的经验。

### 3.7 生产级能力

**面试官可能问：** "这个项目离生产还差什么？"

- **已有的**：FastAPI + SSE 流式推送（事件流实时推送每个阶段的状态）、`--consistency N` 共识模式（多跑一次取综合结果）、ResearchMemory 跨运行持久化
- **可改进的**：并发请求队列、更细粒度的权限控制、用户反馈回路、多语言报告

## 四、工作经历深挖：AskTable

### 4.1 公司背景

**AskTable（察言观数）** 是一个企业级 AI 原生数据智能平台，用户通过自然语言与数据库交互，LLM Agent 完成 SQL 生成、数据分析与可视化。

### 4.2 多供应商 LLM 集成层

**面试官可能问：** "为什么需要多供应商？"

- 实际原因：不同的 LLM 在不同任务上有各自的优势。DeepSeek 性价比高适合批量推理，GPT-4 在复杂 NL2SQL 上更准，Claude 在长上下文处理上更好。统一层让系统能按任务路由到最优模型。
- 设计要点：统一的 `LLMClient` Protocol 定义 `complete(prompt) -> str` 接口，每个供应商实现自己的 adapter。业务代码只依赖 Protocol，不感知具体供应商。
- 成果：新增供应商只需写一个 adapter 文件，不需要修改任何业务代码。

### 4.3 NL2SQL

**面试官可能问：** "NL2SQL 方案怎么做的？"

- 核心是 Schema Linking + RAG 增强。用户问自然语言问题后，系统先做表的召回（从 Qdrant 向量库中检索相关表的 Schema），把 Schema 注入 prompt，再让 LLM 生成 SQL。
- 挑战：企业库有几百张表，不能全量塞进 prompt。Schema Linking 的质量直接影响 SQL 生成的准确性。
- 改进：对常见查询做了模板匹配和 few-shot 示例优化。

### 4.4 Agent 工程化

**面试官可能问：** "Agent 取消机制怎么实现的？"

- 作为生产服务，用户取消查询是常见操作。实现方案是基于 asyncio 的任务取消——每个 Agent 执行任务是一个 asyncio.Task，取消时调用 `task.cancel()`，内部捕获 `asyncio.CancelledError` 做清理（关闭数据库连接、释放搜索配额）。
- 并发控制：使用 asyncio.Semaphore 限制同一用户的并行查询数。

**追问 1：** "SSE 流式推送怎么做？"
- FastAPI 的 StreamingResponse + EventSourceProtocol，Agent 的每个状态变更（planning → searching → synthesizing → done）作为一个 server-sent event 推送，前端通过 EventSource API 接收并展示实时进度。

## 五、技术面试题集

### 5.1 AI Agent 方向

**Q1: LangGraph 的 StateGraph 你怎么理解？**
> StateGraph 的核心是把 Agent 执行建模为一个**状态机**：每个节点是 `(State) -> State` 的纯函数，边定义了节点间的数据流和条件路由。好处是：
> 1. 状态变化可追踪——每一步输入输出都是结构化的
> 2. 支持条件路由——比如 review 分数 < 70 时走重写分支
> 3. 可组合——Pipeline、Multi-Agent、ReAct 三种模式都基于同一套 State/Node 构建

**Q2: 你怎么保证 LLM 输出的可靠性？**
> 我的方法分三层：
> 1. **输入层**（Prompt 工程）：结构化 prompt + Pydantic 输出约束 + 上下文压缩
> 2. **执行层**（验证回环）：引用 7 维校验 + 自动重写 + FACT 真实性验证
> 3. **评估层**（离线度量）：15 个确定性指标衡量每次运行质量
>
> 核心原则：**不信任 LLM 的输出格式和事实一致性，用代码验证替代 prompt 期望。**

**Q3: AI Agent 的核心挑战有哪些？**
> 1. **规划可靠性**：LLM 能否把复杂问题合理分解为可执行的子任务
> 2. **工具使用**：LLM 是否准确选择工具、传递正确参数、解析工具输出
> 3. **记忆管理**：长对话/长研究中的上下文管理，哪些信息保留、哪些丢弃
> 4. **错误恢复**：工具调用失败、中间结果不符合预期时的优雅降级
> 5. **评估**：如何衡量 Agent 做得好不好——这是最难的

**Q4: Function Calling / Tool Calling 怎么设计的？**
> 问这个可以结合项目中 `tools/` 目录的实现。核心是：
> - Tool 注册：统一的 tool registry，每个 tool 定义 name、description、parameters（JSON Schema 格式）
> - Tool 调用循环：LLM 返回 tool_calls → 根据 name 查找 registry → 执行 → 结果返回给 LLM
> - 错误处理：工具超时/异常时返回结构化错误信息，LLM 决定重试或跳过

### 5.2 系统设计方向

**Q5: 设计一个高可用的 AI Agent 服务，你会怎么设计？**
> 要点：
> - **API 层**：FastAPI / gRPC 网关，请求验证 + 限流 + 鉴权
> - **任务队列**：Celery / RabbitMQ 处理异步长任务，Worker 池执行 Agent
> - **状态持久化**：Redis 存会话状态 + PostgreSQL 存结果（支持 ResearchMemory）
> - **监控**：LangSmith 追踪 + Prometheus 指标 + Sentry 错误聚合
> - **降级策略**：LLM API 不可用时缓存回退，搜索失败用缓存结果

**Q6: 如何处理 LLM 的幻觉问题？**
> 从你项目的经验出发：
> 1. **引用约束**：强制 LLM 只引用已提供的信息源（`[N]` 校验）
> 2. **交叉验证**：多来源独立印证，降低单一来源幻觉的影响
> 3. **FACT 验证**：爬取原文验证声明是否被源内容支持
> 4. **不确定性量化**：对低置信度的声明在报告中标注

### 5.3 工程化方向

**Q7: 说说你的测试策略？**
> - 离线测试：FakeLLMClient + FakeSearchClient，不依赖外部 API，<2s 跑完 195 个测试
> - 集成测试：少数端到端测试验证节点间的数据流
> - 评估维度：15 个确定性指标每次运行自动记录
> 关键原则：LLM 测试不应该依赖 LLM——测试的是你的代码逻辑，不是模型的行为。

**Q8: 为什么要用 Protocol 做依赖注入？**
> Python Protocol 在测试中的价值最大：接口定义清晰（IDE 补全 + 类型检查）、测试注入 fake 实现不需要 mock 库、Prod/Dev/Test 使用相同的类型签名。在你的项目中，`LLMClient` 和 `SearchClient` 都是 Protocol，`runner.py` 根据环境组装不同的实现。
